This page documents the supported traffic classes that are explicit in the current implementation.

The important point is that not all traffic enters the same pipeline. Some flows are handled by the
control plane directly, some are forwarded through the dataplane under policy, and some are
steered from the dataplane into service-plane runtimes.

## Flow Categories

The runtime supports four primary traffic categories:

1. DNS queries sent to the firewall
2. Firewall-originated upstream DNS traffic
3. Cluster replication and cluster admin traffic
4. Client traffic evaluated by the dataplane under policy

## DNS Queries Targeted At The Firewall

This is the hostname-policy ingress path.

Shape:

- client source IP -> firewall management IP:53
- UDP or TCP
- handled by the control-plane DNS proxy

Processing steps:

1. The DNS proxy parses the client query
2. It evaluates the source IP and hostname against the compiled `DnsPolicy`
3. Denied queries are answered locally with `NXDOMAIN`
4. Allowed queries are forwarded to configured upstream resolvers
5. Successful IPv4 answers populate the dynamic allowlist and DNS cache

This flow never goes through the normal packet-policy fast path as a generic application stream.
It is a dedicated control-plane service.

## Firewall-Originated Upstream DNS Traffic

Once a client DNS query is allowed, the firewall itself becomes the DNS client.

Shape:

- firewall -> configured `--dns-upstream` addresses
- destination port `53`
- current upstream forwarding logic uses UDP

This traffic is generated by the control plane, not by the protected workload.

From the architecture point of view, that matters because the firewall is not merely relaying raw
packets. It is making a policy decision, originating a new DNS request, validating the upstream
response, and then translating the result into allowlist state.

## Cluster Replication And Cluster Admin Traffic

When cluster mode is enabled, firewall nodes exchange their own east-west traffic.

There are two distinct cluster paths:

- main cluster RPC traffic on the cluster bind address
- join/bootstrap traffic on the join bind address

Main cluster RPC traffic carries:

- Raft replication
- policy admin RPCs
- auth admin RPCs
- integration admin RPCs
- optional wiretap stream fan-out

Join traffic exists only for node enrollment and bootstrap before the joining node has normal
cluster state.

This traffic is firewall-originated control-plane traffic. It is not part of the protected client
data path.

## Policy-Governed Dataplane Traffic

This is the general client or workload traffic path.

Shape:

- source IP belongs to a source group
- traffic enters the dataplane I/O path
- a compiled `PolicySnapshot` decides allow, deny, or service handling

Common dataplane processing concerns:

- source-group lookup
- rule priority evaluation
- NAT
- optional overlay encapsulation
- flow tracking
- audit and wiretap emission

### Normal Allow Or Deny Path

For ordinary L3/L4/ICMP and metadata-only TLS traffic:

1. The dataplane evaluates the packet against the compiled snapshot
2. If denied, it fails closed in the dataplane path
3. If allowed, it forwards the traffic directly
4. NAT and overlay processing are applied as required by the runtime configuration

### DNS-Target Exception

Some traffic that looks like generic client traffic is deliberately diverted away from the normal
packet path.

If all of these are true:

- destination port is `53`
- protocol is TCP or UDP
- source IP is internal
- destination IP is one of the configured DNS target IPs

the dataplane treats the flow as DNS-intercept traffic and hands it to the DNS service path rather
than treating it as an arbitrary external egress flow.

### TLS-Intercept Exception

`tls.mode: intercept` rules create another service-plane detour.

For such a flow:

1. The dataplane still performs the initial source, destination, protocol, and port matching
2. If the matching rule requires intercept mode, policy evaluation marks the flow as requiring
   service handling
3. The flow is steered to the TLS intercept runtime
4. The control plane terminates client TLS and evaluates HTTP policy

This is how the repository preserves the boundary that higher-layer parsing belongs outside the
dataplane.

## DPDK Versus Software Path

The supported traffic categories are the same in software and DPDK mode, but the mechanics differ.

Software mode:

- uses the software adapter path
- relies on kernel steering for interception service handoff

DPDK mode:

- uses the DPDK adapter path
- can bootstrap dataplane addressing through DHCP or IMDS fallback
- uses the service-lane and shared intercept demux path instead of kernel steering

The logical flow categories stay the same even when the I/O implementation changes.

## What Is Not A Supported Flow Category

The implementation does not model these as first-class built-in traffic categories:

- generic control-plane DNS parsing inside the dataplane
- arbitrary non-HTTP application inspection after TLS termination
- unmanaged direct external DNS from clients to any destination without DNS-target interception

Those are useful boundaries because they explain why the repository is organized the way it is.

## Practical Reading Guide

If you are trying to understand a packet or request path:

1. Decide whether the traffic is control-plane service traffic or client dataplane traffic
2. Check whether it is one of the two service-plane detours: DNS-target or TLS-intercept
3. If not, treat it as ordinary dataplane policy enforcement
4. If cluster mode is enabled, separately account for the firewall's own east-west replication
   traffic
