This page explains the behavior of the firewall policy model. It is about evaluation semantics and
operator intent, not just field inventory. For the raw request and schema shape, see the
Configuration Schema reference.

## Mental Model

A policy has three layers:

1. A top-level policy mode
2. A set of source groups
3. A set of ordered rules inside each source group

At runtime, the control plane compiles the configuration into two related products:

- a dataplane packet policy for L3, L4, ICMP, and TLS decisions
- a DNS policy for hostname-based allow and deny decisions

Those two products share the same source-group definitions but do not behave identically in every
edge case.

## Top-Level Policy Mode

Each stored policy record carries a top-level mode:

- `disabled`
- `audit`
- `enforce`

Semantics:

- `disabled`
  The policy record exists in storage, but it is not applied as the active dataplane policy.
- `enforce`
  Normal enforcement mode. A deny decision remains a deny decision.
- `audit`
  The compiled policy is still evaluated, but deny decisions are converted to allow at the final
  enforcement step.

That last point matters: top-level `audit` mode does not mean “ignore the policy.” It means “run
the policy logic, but pass traffic instead of enforcing denies.”

## Source Groups

Source groups are the unit that binds traffic sources to policy intent.

```json
{
  "id": "branch-office",
  "priority": 100,
  "sources": {
    "cidrs": ["10.10.0.0/16"]
  },
  "default_action": "deny",
  "rules": []
}
```

Each source group contains:

- an `id`
- an optional `priority`
- a required `sources` block
- an optional `default_action`
- zero or more `rules`

### Source Membership

The current model supports these source selectors:

- static CIDRs
- static individual IPs
- Kubernetes dynamic selectors via an integration

The control plane rejects a source group whose effective source set is empty.

### Source Group Ordering

Source groups are sorted by ascending priority before evaluation. If `priority` is omitted, the
compiler falls back to the group’s list position.

This gives two practical rules:

- lower numeric priority wins over higher numeric priority
- if priorities are omitted, earlier list entries win

## Rules

Rules live inside a source group and are also sorted by ascending priority before evaluation.

```json
{
  "id": "allow-web",
  "priority": 10,
  "action": "allow",
  "mode": "enforce",
  "match": {
    "proto": "tcp",
    "dst_ports": [443]
  }
}
```

Each rule has:

- an `id`
- an optional `priority`
- an `action` of `allow` or `deny`
- a per-rule `mode`
- a required `match` block

### First-Match Wins

Within a matching source group, rules are evaluated in priority order and the first matching rule
terminates evaluation.

That applies to:

- packet policy evaluation
- DNS policy evaluation
- TLS metadata and intercept matching

## Rule Mode vs Policy Mode

There are two distinct mode concepts:

- top-level policy `mode`
- per-rule `mode`

Per-rule `mode` can be:

- `enforce`
- `audit`

Semantics:

- `enforce` rules participate in normal dataplane enforcement
- `audit` rules are excluded from the enforce path and are only visible through audit evaluation

This separation is important:

- a top-level `audit` policy still evaluates its `enforce` rules, but deny outcomes are converted
  to allow at the end
- a rule with `mode: audit` does not enforce even when the policy is in `enforce` mode

In other words:

- top-level policy mode changes how final enforcement behaves
- rule mode changes which rule set a decision belongs to

## Packet Evaluation Order

For non-DNS packet policy, the runtime evaluation flow is:

1. Find source groups whose `sources` contain the packet source IP
2. Walk those source groups in priority order
3. Inside each matching source group, walk rules in priority order
4. Return on the first matching rule
5. If no rule matched and the source group has `default_action`, return that action
6. If no matching source group produced a decision, fall back to top-level `default_policy`

That means `default_action` is stronger than the top-level `default_policy` once a source group
claims the source IP.

## DNS Evaluation Semantics

DNS policy uses the same source-group structure, but its fallback behavior is stricter.

When a source IP belongs to a source group:

- matching DNS rules are evaluated in priority order
- the first match wins
- if no DNS rule matches, the source group is treated as default-deny for enforcement

This is a deliberate behavior difference from packet policy. DNS does not use `default_action` in
the same way the dataplane rule engine does. Once a source group claims a source IP, absence of a
matching DNS rule means deny.

Practical consequence:

- if you want hostname-based access for a source group, you need explicit DNS allow rules for the
  hostnames you intend to permit

## Defaults And Fallbacks

There are three relevant fallback points:

- rule hit
- source-group `default_action`
- top-level `default_policy`

For packet policy:

- if a rule matches, its action wins
- else if the source group has `default_action`, that wins
- else the evaluator continues to later matching source groups
- if no source group yields a decision, `default_policy` is used

For DNS policy:

- if a DNS rule matches, its action wins
- else a matching source group defaults to deny during enforcement evaluation
- if no source group contains the source IP, the result is also deny during enforcement evaluation

## Match Semantics

The `match` block is additive. A rule only matches when all populated selectors match together.

Current selector families:

- destination CIDRs and destination IPs
- protocol
- source and destination ports
- ICMP type and code filters
- DNS hostname regex
- TLS metadata matchers
- TLS intercept HTTP request and response matchers

### Protocol Semantics

`proto` accepts either a string or a protocol number. Common string forms are:

- `tcp`
- `udp`
- `icmp`
- `any`

If `proto` is omitted, the compiler treats it as `any`.

### Port Semantics

Ports can be expressed as:

- a single integer
- a range string such as `"1000-2000"`

If a port list is empty, that dimension is treated as unconstrained.

### ICMP Semantics

ICMP matching has extra validation rules:

- ICMP type and code filters require `proto` to be `icmp` or `any`
- ICMP rules cannot also constrain `src_ports` or `dst_ports`
- if an ICMP rule sets `proto: icmp` and leaves both type and code empty, the compiler injects a
  default set of common ICMP controls

That default set is:

- types: `0`, `3`, `11`
- codes: `0`, `4`

## DNS Hostname Semantics

`dns_hostname` is compiled as a case-insensitive regex matcher. Hostnames are normalized before
matching, including trailing-dot handling.

Practical consequences:

- case differences do not matter
- invalid regex patterns are rejected at write time
- empty regex strings are rejected
- explicit priority is the way to express “specific deny before broad allow”

Example:

```yaml
source_groups:
  - id: dns
    sources:
      ips: ["192.0.2.2"]
    rules:
      - id: deny-specific
        priority: 0
        action: deny
        match:
          dns_hostname: '^bar\.allowed$'
      - id: allow-wildcard
        priority: 1
        action: allow
        match:
          dns_hostname: '.*\.allowed$'
```

Here, `bar.allowed` is denied while `baz.allowed` is allowed.

## TLS Policy Semantics

TLS matching lives under `match.tls` and has two major modes:

- `metadata`
- `intercept`

### Metadata Mode

Metadata mode evaluates TLS properties such as:

- SNI
- SAN
- CN
- certificate fingerprint
- trust anchors

If a metadata-based TLS rule matches, its allow or deny action is returned.

If the TLS observation required for the match is not available yet, the dataplane can return a
`PendingTls` decision while waiting for enough information.

### Intercept Mode

Intercept mode is for HTTP-aware policy on top of TLS interception.

Important constraints:

- `tls.http` is only valid when `tls.mode` is `intercept`
- `tls.mode: intercept` requires actual HTTP constraints under `tls.http`
- intercept mode cannot be combined with metadata-oriented TLS name matchers

The intercept path is intentionally stricter because it changes how traffic is handled at runtime.

## HTTP Matchers Inside TLS Intercept

When using `tls.http`, the model currently supports:

- request host matching
- request method matching
- request path matching
- request query matching
- request header matching
- response header matching

Method names are normalized to uppercase during compilation.

This is not a generic HTTP policy DSL. It is specifically the HTTP policy surface supported by the
TLS intercept runtime today.

## Kubernetes-Derived Sources

Kubernetes source selectors let a source group track pod or node IPs dynamically through an
integration.

Two rules matter here:

- exactly one of `pod_selector` or `node_selector` must be present
- the referenced `integration` must exist when the policy is written through the HTTP API

This keeps source discovery in the control plane and out of the dataplane, which matches the
repository’s architectural split.

## Writing Policies Safely

A few practical guidelines follow directly from the current evaluator:

- Use explicit low priorities for specific exceptions before broad rules.
- Use `default_action` when you want a source group to own its traffic posture.
- Use top-level `default_policy` as the last resort, not as a substitute for source-group intent.
- Treat DNS allowlists as explicit allowlists. Once a source group owns a source IP, unmatched DNS
  lookups are denied in enforcement mode.
- Use rule-level `mode: audit` for exploratory rules without changing the whole policy’s runtime
  behavior.
- Use top-level `mode: audit` when you want to observe the full policy without enforcing denies.
