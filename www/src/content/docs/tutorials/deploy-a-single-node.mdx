This tutorial covers the smallest supported deployment shape:

- one booted Neuwerk appliance image
- one running Neuwerk process
- no cluster replication
- one management interface
- one dataplane interface

If you are unsure whether you should be thinking in terms of image requirements or binary
requirements, read [Requirements](/docs/deployment/requirements) first.

Use the shipped appliance image when possible. The image is the primary distribution artifact and
already carries the matching DPDK runtime and packaged service assumptions.

You will finish with:

- the appliance image booted and the Neuwerk process running
- the management HTTPS endpoint answering
- the metrics endpoint answering
- a short-lived admin token minted for first login
- enough context to add your first policy safely

## Before You Start

Have these values ready:

- a booted Neuwerk appliance VM with console or SSH access
- a management interface with an IPv4 address
- a separate data-plane interface
- at least one DNS target IP that clients will send DNS traffic to
- at least one upstream DNS resolver

Important constraints:

- the management and data-plane interfaces must be different
- DNS policy depends on the Neuwerk receiving client DNS traffic
- the smallest deployment shape still runs both the control plane and the dataplane

## Start The Neuwerk

The appliance image should be configured so that the packaged Neuwerk service starts with a runtime
shape equivalent to:

```text
neuwerk \
  --management-interface mgmt0 \
  --data-plane-interface data0 \
  --dns-target-ip 10.0.0.53 \
  --dns-upstream 1.1.1.1:53
```

What this does:

- enables the management HTTPS listener on the management IP
- enables the metrics listener on the management IP
- enables the DNS proxy on the management IP at port `53`
- leaves cluster replication disabled

Use this command shape as the reference contract for a single node. If you are using the shipped
image, do not rebuild or repackage the binary just to follow this guide.

## Verify The Process

Check that the management process is alive:

```text
GET /health
```

Then confirm that metrics are exposed:

```text
GET /metrics
```

If you already have management credentials, also check readiness:

```text
GET /ready
GET /api/v1/stats
```

Readiness matters more than liveness. A healthy node is up; a ready node is able to enforce
traffic with the dataplane, DNS proxy, and service plane in a usable state.

## Mint A First Admin Token

Once the node is healthy, mint a short-lived admin token locally from the appliance:

```bash
umask 077
sudo neuwerk auth token mint \
  --sub operator@example.com \
  --roles admin \
  --ttl 30m \
  --http-tls-dir /var/lib/neuwerk/http-tls > neuwerk-admin.jwt
```

Use the minted token to verify access:

```bash
export NEUWERK_TOKEN="$(cat neuwerk-admin.jwt)"
curl -sk \
  -H "Authorization: Bearer $NEUWERK_TOKEN" \
  https://<management-ip>:8443/api/v1/auth/whoami
```

That local CLI bootstrap is the preferred first-admin flow for single-node deployments because it
requires only operator access to the node itself.

## Confirm The Listener Layout

Unless you override the defaults, a single node listens on:

- `management-ip:8443` for the web UI and management API
- `management-ip:8080` for Prometheus metrics
- `management-ip:53` for DNS proxy traffic

The node does not start cluster listeners in this mode.

## What To Check If Startup Fails

Start with these questions:

- does the management interface have an IPv4 address
- are the management and data-plane interfaces different
- did you provide at least one `--dns-target-ip`
- did you provide at least one `--dns-upstream`

If the process is alive but not ready, continue with [Troubleshoot Enforcement](/docs/operations/troubleshooting).
